Step 1 · Prerequisites

Validate that your tenant has the licenses, admin roles, baseline configuration and the leadership constructs (AI Council, Responsible AI principles) required for a Copilot Chat rollout. Complete this checklist before moving on.

30 min
🪪

1.1 · Eligible licenses

Copilot Chat is included with any qualifying Microsoft 365 or Office 365 subscription that grants users a signed-in work or school identity. Confirm seats in the Microsoft 365 admin center under Billing → Licenses. For paid Microsoft 365 Copilot, plan E3 + SAM (Foundational) or E5 + SAM (Optimized) — you will choose the path in Step 2.

👤

1.2 · Admin roles you need

Assign least-privilege roles in Microsoft Entra: Global Reader (audit), Compliance Administrator (Purview), SharePoint Administrator (SAM, RAC), Application Administrator (Integrated apps), Reports Reader (Copilot Dashboard).

🌐

1.3 · Tenant baseline

Verify: tenant is in worldwide multi-tenant or supported sovereign cloud, Modern Authentication enabled, OneDrive provisioned for target users, SharePoint search service healthy, Audit (Standard or Premium) on.

📊

1.4 · Optimization Assessment

Run the Microsoft 365 Copilot Optimization Assessment (26 questions, ~30 min, self-service). It quantifies your current data-security posture and points to the right remediations. Carry the score and gap list into Step 2.

Open Solution Assessments →

🧭

1.5 · AI Council & Responsible AI

Stand up an AI Council (executive sponsor, IT, Security, Compliance, Legal, HR, Comms, business reps) and adopt Microsoft’s Responsible AI principles (fairness, reliability & safety, privacy & security, inclusiveness, transparency, accountability). The Council owns approvals across Steps 2–8.

Step 2 · Run the Copilot readiness assessment

Complete this checklist to baseline your tenant before broad deployment. Every task below is a customer-runnable action in the Microsoft 365 admin center, Microsoft Purview portal or Microsoft Defender. Save the outputs — they are the input for Steps 3–6.

1–2 weeks
📋

2.1 · Open the Copilot setup guide

Sign in to the Microsoft 365 admin center → Setup → Microsoft 365 Copilot setup guide and run the built-in readiness checks. The guide inspects your tenant configuration and lists action items in plain language.

🛡️

2.2 · Review Microsoft Secure Score

In Microsoft Defender → Secure Score, filter the improvement actions by Identity, Data and Apps. Capture your current score and target a baseline before opening Copilot Chat to the pilot.

📊

2.3 · Check the Copilot dashboard

In M365 admin center → Reports → Microsoft 365 Copilot (and Viva Insights → Copilot Dashboard), confirm the dashboard is enabled and reporting is flowing. You will need it for Step 7 measurement.

🏷️

2.4 · Inventory sensitivity labels

In Microsoft Purview → Information Protection → Labels, list the labels currently published. Record adoption rate (per Activity explorer). If coverage is below 30%, plan auto-labeling in Step 3.

🔎

2.5 · Run the oversharing baseline

In SharePoint admin center → Reports → Data access governance, generate the oversharing baseline report (SharePoint Advanced Management). Export to CSV. Sites flagged here are remediated in Step 4.

🌐

2.6 · Confirm app & network readiness

In Microsoft 365 Apps admin center → Inventory, verify that ≥ 95% of devices are on Current Channel. Validate egress to *.cloud.microsoft and *.office.com using the Microsoft 365 connectivity test.

Step 3 · Data governance with Microsoft Purview

Copilot honours your existing sensitivity labels, DLP policies and retention rules. Establish them now so Copilot derivatives inherit protection automatically. Every task below is performed in the Microsoft Purview portal by a Compliance Administrator or Information Protection Administrator.

1–3 weeks
🏷️

3.1 · Sensitivity labels

Define a 4–5 tier taxonomy (Public, Internal, Confidential, Highly Confidential). Publish via label policies, then enable auto-labeling for the highest tiers so Copilot honours classification on first contact.

🛑

3.2 · DLP for Copilot

Target the Microsoft 365 Copilot DLP location. Block or warn when prompts or grounded content match sensitive types (PII, financial, source code, customer data).

♻️

3.3 · Retention & deletion

Publish retention labels and policies. Decommission inactive Teams, Groups and sites so Copilot does not surface stale or obsolete content in answers.

Detailed implementation tasks — click each card to expand. Each lists the portal path, the exact configuration, verification steps and the output to capture in your readiness library.

3.1.1Design the label taxonomy
  • Recommended tiers: Public (no encryption), General/Internal (no encryption, header), Confidential (encryption + watermark, scoped to org), Confidential / Specific people (encryption + user-defined recipients), Highly Confidential (encryption + DRM + content marking).
  • Sub-labels: consider sub-labels per business unit (e.g. Confidential / Finance, Confidential / HR) when departmental controls are required.
  • Decision required: approval workflow for label downgrade (justification mandatory? approver group?).
  • Output: a one-page taxonomy diagram with tier, color, encryption mode, content marking, scope and example file types per tier. Save as 03-01_label-taxonomy.pdf.
  • Approval: CISO or delegate, Legal/Privacy, Records Manager.
3.1.2Create & publish labels · Foundational
  • Where: Microsoft Purview → Information Protection → Labels → Create a label.
  • For each label, configure: display name, description for users, description for admins, color, scope (Files & Emails / Groups & Sites / Schematized data / Meetings), encryption (mode, permissions, content expiration, offline access), content marking (header, footer, watermark), auto-labeling client-side conditions, auto-labeling service-side conditions, protect Teams & Groups settings.
  • Microsoft 365 Copilot label settings (key):
    • Block Copilot access: on a sensitivity label, enable Use Microsoft Purview to control Copilot processing → Prevent Microsoft 365 Copilot from processing files with this label. Use this on the most sensitive tiers (e.g. Highly Confidential / Restricted) to keep specific content out of Copilot grounding entirely.
    • Output inheritance: Copilot responses and Copilot-created documents inherit the sensitivity label and protections of the source content, so encryption, watermarks and DRM persist on AI-generated derivatives without extra configuration.
  • Publish: Information Protection → Label policies → Publish labels. Choose target users/groups, set the default label per location (Email, Files, Sites, Power BI, Meetings), enable require justification on downgrade, enable require label on save for the pilot.
  • Verification: log in as a pilot user; confirm the labels appear in Word, Excel, PowerPoint and Outlook within 24 h. Use Activity explorer to confirm label-applied events.
  • Target: 100% of pilot users see the published labels within 48 h of publication.
  • Feeds: Step 4.F.6 (oversharing remediation reuses these labels), Step 4.F.7 (DLP simulation conditions match by label).
3.1.3Auto-labeling — client-side first, then service-side · Foundational
  • Where (client-side): on each label, configure Auto-labeling for files and emails → client-side conditions. The Office app recommends or applies the label on save based on local content inspection.
  • Where (service-side): Information Protection → Auto-labeling policies → Create an auto-labeling policy. The service inspects content already at rest in SharePoint, OneDrive and Exchange.
  • Microsoft-recommended sequence:
    1. Start with client-side recommendations to teach users which content needs which label and capture telemetry on suggestion-acceptance rate.
    2. Move to client-side automatic apply for high-confidence patterns (low false-positive risk).
    3. Extend to service-side auto-labeling policies for at-rest coverage and to retro-label legacy content.
  • Recommended starting policies: auto-label any file containing ≥ 1 highly sensitive type (e.g. credit card, national ID, secret keys) as Highly Confidential; auto-label files in named-entity HR or Finance sites as Confidential.
  • Run modes: always start service-side policies in Simulation for at least 7 days. Review matches in Auto-labeling activity; tune conditions until false-positive rate < 5%; then switch to Turn on policy.
  • Locations: SharePoint Online sites, OneDrive accounts, Exchange Online (limited to specific senders/recipients during pilot).
  • Output: simulation report screenshot + decision log of accepted/rejected matches. Save as 03-13_autolabel-simulation.csv.
3.1.4Verify label coverage with PowerShell
  • Module: Install-Module -Name ExchangeOnlineManagement (includes the Security & Compliance cmdlets via Connect-IPPSSession).
  • Run: 
    Connect-IPPSSession
Get-Label | Select-Object DisplayName, Priority, ContentType, IsValid | Format-Table -AutoSize

  • Cross-check with deliverable 02-04 from Step 2: coverage % should be trending up week over week.
3.1.5Apply site sensitivity labels — mark sites “Private” · Foundational
  • Where: SharePoint admin center → Sites → Active sites → select site → Settings → Sensitivity; or Microsoft Purview → Information Protection → Label policies with Groups & sites scope.
  • What it does: a site sensitivity label sets the site privacy and external-sharing posture in one move — e.g. mark business-critical sites as Private with Block external sharing + Members only. This is the simplest lever to limit org-wide exposure before Copilot starts grounding.
  • Recommended baseline: apply Confidential site label (Private, members only) to executive, HR, finance, legal and M&A sites; apply Internal site label to general business sites; allow Public only for explicit collaboration spaces.
  • Verify: the site card in SharePoint admin shows the correct label; external-sharing controls reflect the label policy; Copilot answers do not surface content from those sites for non-members.
  • Output: site-label assignment list. 03-15_site-labels.csv.
  • Feeds: Step 4.F.5 (RAC layered on top of site labels for the highest tier).
3.1.6Default site library labels — protect new docs by default · Foundational
  • Where: on each SharePoint site library → Settings → Library settings → Default sensitivity label (or via Purview label policy).
  • What it does: every new file added to the library, and every existing unlabeled file modified there, is auto-labeled with the chosen sensitivity label. Closes the “forgotten label” gap that allows oversharing.
  • Recommended use: set Confidential as default on libraries inside Confidential-labeled sites; set Highly Confidential default in Restricted libraries.
  • Verify: upload an unlabeled file to a configured library; confirm the label appears within minutes; confirm the file then participates in DLP and RAC enforcement.
  • Output: default-label assignment per library. 03-16_default-library-labels.csv.
3.1.7Use Purview Content Explorer to map SITs to sites · Foundational
  • Where: Microsoft Purview portal → Information Protection → Content explorer.
  • What it does: shows which SharePoint sites, OneDrive accounts and mailboxes contain documents matching specific sensitivity labels or sensitive information types (SITs).
  • Use it to:
    • Identify high-SIT-density sites that should be labeled Confidential or Highly Confidential.
    • Validate that auto-labeling policies (3.1.3) are reaching the expected content.
    • Cross-reference with deliverable 02-05 (oversharing baseline) to find sites that combine high-SIT density and broad sharing — the highest priority for Step 4 remediation.
  • Permissions required: Content Explorer Content Viewer or Content Explorer List Viewer role group (separate from Compliance Admin to enforce least privilege).
  • Output: top-50 high-density sites list. 03-17_content-explorer-sites.csv.
3.2.1Create the DLP policy for Microsoft 365 Copilot · Foundational
  • Where: Microsoft Purview → Data Loss Prevention → Policies → Create policy.
  • Template: start from the built-in Microsoft 365 Copilot template (or Custom if not yet rolled out in your region).
  • Locations to enable: Microsoft 365 Copilot (the new location) plus SharePoint sites, OneDrive accounts, Exchange Email and Teams chat & channel messages for end-to-end protection.
  • Conditions: match content with sensitivity labels Confidential and Highly Confidential, or content containing sensitive info types (Credit Card Number, Social Security Number, IBAN, secret keys, customer-list classifier).
  • Actions: for the pilot, set to Audit only with policy tip; for production, set to Block with override for Confidential and Block for Highly Confidential.
  • User notifications: enable policy tip in Copilot Chat experience explaining why the response was redacted.
  • Output: screenshot of policy summary + DLP rule names list. Save as 03-21_dlp-copilot-policy.pdf.
3.2.2Test the DLP policy end-to-end
  • Test cases (run as a pilot user):
    • Prompt that returns a known Confidential document → expect policy tip in Copilot Chat.
    • Prompt that returns a Highly Confidential document → expect blocked with explanation.
    • Prompt that asks Copilot to summarise content containing a fake credit-card number → expect redaction.
    • Prompt grounded in a public site → expect normal answer (no false positive).
  • Verification location: Purview → Activity explorer — filter by DLP rule matched and confirm the matches per test.
  • Output: test-case matrix with expected vs. actual result. Save as 03-22_dlp-test-results.xlsx.
  • Pass criteria: 100% of expected blocks occurred; false positive rate on neutral prompts < 2%.
3.2.3Configure Adaptive Protection · Optimized path
  • Where: Purview → Insider Risk Management → Adaptive Protection.
  • What it does: dynamically tightens DLP and conditional access for users whose insider-risk score increases (e.g. departing employees, recent investigations).
  • Recommended scope: apply to Copilot DLP location so risky users see stricter Copilot guardrails automatically.
  • Output: policy export + decision log. Save as 03-23_adaptive-protection.pdf.
3.3.1Publish retention labels & policies
  • Where: Purview → Data Lifecycle Management → Microsoft 365 → Retention policies and Retention labels.
  • Recommended retention tiers: Business records (retain 7 years, then review), Working content (retain 3 years, then delete), Transient (delete after 1 year).
  • Locations to cover: SharePoint sites, OneDrive accounts, Microsoft 365 Groups, Teams chat, Teams channel messages, Exchange mailboxes.
  • Configuration tip: apply retention labels via auto-labeling rules (file type, sensitivity label match) to avoid relying on user action.
  • Output: retention matrix (label → trigger → period → disposition action). Save as 03-31_retention-matrix.xlsx.
3.3.2Decommission inactive workspaces
  • Where: SharePoint admin center → Reports → Site usage and Microsoft 365 admin center → Active users / Groups → Inactive groups report.
  • Criteria: Teams/Groups with no activity for 180 days, sites with no document edits for 365 days, OneDrive of departed users older than HR retention period.
  • Process: notify owner via automated email (Power Automate flow), wait 30 days, archive (read-only), wait 60 days, delete; for sites under retention, hold deletion until retention period elapses.
  • Output: decommission backlog (CSV) with site URL, owner, last activity, recommended action and decision date. Save as 03-32_decommission-backlog.csv.
  • Why it matters for Copilot: stale sites pollute search and grounding; retiring them measurably improves Copilot answer relevance.
3.4Communication Compliance for Copilot interactions · Optimized path
  • Where: Microsoft Purview portal → Communication Compliance → Policies → Create policy.
  • Why for Copilot: in regulated industries (finance, healthcare, government, defense), prompts and responses are communications subject to supervisory rules. Comm Compliance lets you proactively detect risky AI-use patterns and route them to a reviewer queue.
  • Required licensing: Microsoft 365 E5 / E5 Compliance (this is an Optimized-path control — not in scope for Foundational deployments unless purchased standalone).
  • Locations / users: scope to the pilot population first; expand to all licensed users at scale.
  • Conditions to detect (recommended starting set):
    • Built-in Sensitive information classifier (PII, financial, health, regulated identifiers)
    • Built-in Threats, Harassment and Discrimination classifiers
    • Built-in Regulatory compliance classifier (insider trading, money laundering, gifts & entertainment) for finance
    • Custom classifier or keyword list for your industry-specific risk vocabulary
  • Reviewer workflow: assign a Compliance reviewer group, define escalation path (reviewer → lead → legal), set SLA (e.g. triage within 24 h, decision within 5 days).
  • Privacy posture: by default, reviewers see pseudonymised user identities; switch to identified mode only for confirmed cases per your works-council / privacy policy.
  • Verify: trigger a known sensitive prompt as a pilot user; confirm the policy match appears in Communication Compliance → Alerts within 1 hour.
  • Output: policy export (JSON) + reviewer onboarding note + first-month alerts triage log. 03-41_commscompliance-policy.pdf · 03-42_commscompliance-triage.xlsx.
  • Feeds: Step 6.4 (Comm Compliance integration with Copilot Control System) and Step 7 (review alerts during pilot).
3.5eDiscovery readiness for Copilot prompts & responses · Optimized path
  • Where: Purview → eDiscovery → Premium.
  • Verify: Audit (Standard or Premium) is enabled tenant-wide; Copilot interactions are stored in the user's Exchange mailbox SubstrateHolds folder and are discoverable.
  • Test: create a test eDiscovery case, search for prompt keywords across the pilot population, confirm hits are returned within SLA.
  • Output: test-case run sheet + screenshot of search results. Save as 03-51_ediscovery-readiness.pdf.

  • Approvers: CISO, Data Protection Officer, Records Manager, Legal/Privacy.

3.7DSPM for AI — data security posture for Copilot · Optimized path
  • Where: Microsoft Purview portal → Data Security Posture Management for AI (DSPM for AI).
  • What it does: gives security and compliance teams a single pane of glass over the data Microsoft 365 Copilot can access, the prompts/responses that contain sensitive content, and policy suggestions to close the gaps.
  • Required licensing: Microsoft 365 E5 / E5 Compliance — Optimized path control.
  • Configure these capabilities:
    1. Run a Copilot data assessment — baseline view of which sites, files and SITs are reachable by Copilot. Re-run monthly during pilot, quarterly afterwards.
    2. Create custom assessments targeted at specific Microsoft 365 locations (executive, finance, M&A) pivoted on the SITs from 3.1.7.
    3. Reports of sensitive data and unprotected files referenced in Copilot interactions — review weekly during pilot.
    4. Act on policy suggestions — DSPM proposes label, DLP and RAC policies tailored to detected risk. Apply via the guided workflow (cross-references Steps 3.1, 3.2 and 4.O.2).
    5. Subscribe to oversharing notifications — route to the Compliance reviewer queue with one-click remediation.
