1. Objective

Apply, in a controlled and progressive manner, the organization's versioning standard to existing document libraries in SharePoint Online, closing the gap between new libraries (which already inherit the tenant policy) and existing libraries (which still carry their original settings).

This is a configuration-only change — no file versions will be deleted. The policy defines limits going forward; any future version trimming would be a separate, independently approved initiative.

2. Current State vs. Target State

Current StateTarget State
New document librariesInherit tenant policy: 50 major versions, no time expirationNo change — already aligned
Existing document librariesRetain original settings (typically 500 major versions or SharePoint default)Standardized to 50 major versions, no time expiration (ExpireVersionsAfterDays = 0)
OneDrive librariesFollow tenant or account-level policyOut of scope — separate initiative
💡
What does ExpireVersionsAfterDays = 0 mean?
A value of 0 means "never expire" — versions are kept indefinitely and are only removed when they exceed the major version count limit. It does not mean "expire immediately".

3. How Version History Limits Work

SharePoint Online applies version history limits through an inheritance model with three levels:

LevelApplies ToBehavior
Organization (Tenant)All new libraries on sites without site-level overridesSet via SharePoint Admin Center or Set-SPOTenant. Default baseline for the entire tenant.
SiteAll new libraries on the site; optionally existing librariesBreaks inheritance from org defaults. Set via Set-SPOSite. Can target new, existing, or both.
LibraryIndividual library onlyBreaks inheritance from site/org. Set via Set-SPOListVersionPolicy or Set-PnPList.

Key Technical Behaviors

  • When using -ApplyToExistingDocumentLibraries, settings on existing libraries are updated asynchronously via a background job that can take up to 24 hours to complete.
  • Setting new limits does not trim existing versions. Versions exceeding the new limit are only removed as new versions are created (rolling window) or through an explicit trim job.
  • Cancelling an in-progress job stops processing of remaining libraries but does not revert changes already applied.
  • Items under retention policies or eDiscovery holds are exempt — versioning limits are ignored while the hold is active.
  • Site-level settings can be managed via PowerShell only — there is no UI for this in the SharePoint Admin Center.

4. Scope

In Scope

  • Existing SharePoint Online sites and document libraries with versioning enabled
  • Applies to SharePoint Online sites only — OneDrive accounts are excluded from this deployment plan and should be handled as a separate initiative if needed
  • Applying the versioning policy to existing libraries via -ApplyToExistingDocumentLibraries
  • Library-level exceptions for libraries with specific business requirements
  • Validating behavior, operational impact, and governance adherence

Out of Scope

  • Deletion of existing versions (trimming) — requires separate approval and What-if analysis
  • OneDrive personal libraries and account-level version settings
  • Changes to retention policies, eDiscovery, sensitivity labels, or compliance settings
  • Changes to permissions, site structure, or information architecture
  • File-type-specific version limits (-FileTypesForVersionExpiration)
🚨
Critical Distinction: Policy Change vs. Trimming
Policy change (this plan): Sets the version limit going forward. Existing versions beyond the limit are only removed as new versions are created. This is reversible — you can change the limit again at any time.

Trimming (NOT this plan): Actively deletes stored versions that exceed the limit. Versions removed by trimming are permanently deleted and bypass the recycle bin. This is irreversible.

7. Assumptions

  • The tenant-level policy is already configured: MajorVersionLimit = 50, ExpireVersionsAfterDays = 0, EnableAutoExpirationVersionTrim = $false
  • The SharePoint Online Management Shell (v16.0.24009.0+) is installed and the executing account has SharePoint Administrator privileges
  • A site inventory with owner, criticality, and storage data is available or will be created before Phase 1
  • Sites under active retention policies, legal holds, or eDiscovery will be identified and excluded prior to execution
  • No trimming will be performed during this deployment — trimming requires a separate approval process
  • Site owners are available for post-change validation within 48 hours of execution

PowerShell

    Select-Object Url,
        EnableAutoExpirationVersionTrim,
        MajorVersionLimit,
        ExpireVersionsAfterDays

Apply Policy to Existing Libraries (Site-Level)

PowerShell
Set-SPOSite -Identity "https://<tenant>.sharepoint.com/sites/<site-name>" `
    -EnableAutoExpirationVersionTrim $false `
    -MajorVersionLimit 50 `
    -ExpireVersionsAfterDays 0 `
    -ApplyToExistingDocumentLibraries

Track Background Job Progress

PowerShell
Get-SPOSiteVersionPolicyJobProgress `
    -Identity "https://<tenant>.sharepoint.com/sites/<site-name>"
# Possible statuses: NoRequestFound | New | InProgress | CompleteSuccess | CompleteWithFailure

Cancel an In-Progress Job

PowerShell
Remove-SPOSiteVersionPolicyJob `
    -Identity "https://<tenant>.sharepoint.com/sites/<site-name>"

            Status = "Success"; Message = "Policy applied"

    catch {
        $results += [PSCustomObject]@{
            Wave = $item.Wave; Site = $site
            Status = "Failed"; Message = $_.Exception.Message

$results | Export-Csv ".\wave-results-$timestamp.csv" -NoTypeInformation

↩️ Rollback Plan

🚨
Critical Distinction — Read Before Proceeding
Policy change = REVERSIBLE — You can change version limits again at any time. Existing versions are NOT deleted when a policy is changed. A rollback simply sets the limits back to the previous value.

Trimming = IRREVERSIBLE — Versions deleted by trimming bypass the recycle bin and CANNOT be recovered by any means (including Microsoft Support). There is no rollback for trimming.

Cancelling an in-progress background job = PARTIAL — Cancelling stops processing of remaining libraries, but does NOT revert changes on libraries already processed. You must explicitly revert those libraries.

Understanding Rollback Behavior

Before executing any rollback, understand these key behaviors:

ScenarioWhat HappensWhat Does NOT Happen
Revert site policy to 500 versionsNew versions will be kept up to the new limit (500). Existing libraries are updated asynchronously.No versions are deleted. Versions already stored are retained.
Clear site override (inherit from tenant)New libraries inherit tenant defaults. Existing libraries keep their current settings.Existing libraries are NOT automatically reverted — they retain the last applied setting.
Cancel background job mid-progressUnprocessed libraries keep their old settings.Already-processed libraries are NOT reverted. You must re-apply to revert them.
Revert a single libraryOnly that library's settings change. No impact on other libraries or the site-level policy.The site-level policy remains unchanged.
Sync library to site policyLibrary inherits the current site-level setting (whatever it is at that moment).Does not sync to tenant policy — only to site-level.
⚠️
Important: Clearing a Site Override
Using -InheritVersionPolicyFromTenant or Set-PnPSiteVersionPolicy -InheritFromTenant only affects new libraries created after the command. It does NOT revert existing libraries to the tenant default. To revert existing libraries, you must either:
— Re-apply the previous policy with -ApplyToExistingDocumentLibraries, or
— Use New-SPOSiteManageVersionPolicyJob -SyncListPolicy to sync all libraries to the site-level setting, or
— Revert libraries individually with Set-SPOListVersionPolicy.

Rollback Decision Flowchart

🔄
Rollback Decision Flow
Step 1: Issue detected → Classify severity (P1 / P2 / P3 / P4)

Step 2 (P1/P2): Pause current wave immediately → Notify stakeholders → Proceed to Step 3
Step 2 (P3/P4): Log issue → Continue wave with monitoring → Investigate in parallel

Step 3: Identify affected site(s) and scope of impact
— How many sites affected?
— How many users impacted?
— Is the background job still running?

Step 4: Is there a background job still in progress?
→ YES: Cancel it first with Remove-SPOSiteVersionPolicyJob (Option F)
→ NO: Proceed to Step 5

Step 5: Is the issue caused by the policy change?
→ YES: Proceed to Step 6
→ NO: Investigate other causes (permissions, tenant health, network); do NOT rollback
→ UNCLEAR: Rollback as precaution, then investigate

Step 6: Determine rollback scope:
→ Entire site: Option A (revert to 500 versions + ApplyToExisting) or Option B (inherit from tenant)
→ Specific libraries: Option D (revert library) or Option E (sync library to site)
→ All libraries need re-sync: Option C (New-SPOSiteManageVersionPolicyJob -SyncListPolicy)

Step 7: Execute rollback → Track the new background job progress

Step 8: Validate rollback:
— Confirm settings restored via Get-SPOSite and Get-PnPList
— Confirm functionality OK (upload, edit, version history, restore)
— Confirm no new errors

Step 9: Record incident:
— Incident details, root cause, affected sites/libraries
— Rollback method used, time to resolve
— Lessons learned and runbook updates

Step 10: Resume wave only after:
— Root cause identified and corrected
— Rollback validated
— Go/No-Go approval from IT Manager

🎯 Final Recommendation & Executive Summary

Recommendation

  1. Do not start with trimming.
  2. First, apply only the versioning policy to existing libraries.
  3. Validate behavior and stability across all phases.
  4. Then, if the objective is to recover storage, open a separate plan for:
    • Impact analysis (What-if)
    • Formal approval
    • Controlled trimming by site or library
💡
Key Takeaway
The initial change is a governance/configuration update (reversible). Trimming is a destructive and permanent action (irreversible). Keeping them separate reduces risk significantly.

Executive Summary

  • POC: Validate the policy application on a single controlled site.
  • POC2 / Test: Validate across multiple library profiles and exception handling.
  • Small Production Test: Confirm behavior in production with low risk and real users.
  • Full Deployment: Apply in waves to all eligible sites with logging, monitoring, and exception management.

The policy can be applied at the site level to existing libraries using -ApplyToExistingDocumentLibraries. Any trimming action should be treated separately, as it removes versions permanently and without recycle bin recovery.
