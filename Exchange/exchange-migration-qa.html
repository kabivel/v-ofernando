Migration Playbook

Exchange On-Premises → Exchange Online

Scope: on-premises source only — does not cover tenant-to-tenant, IMAP, or Google Workspace migrations.

1. Planning & Methods

What migration methods are available for Exchange on-prem to Exchange Online? Planning
  • Cutover migration — All mailboxes moved at once. Limit ~150 mailboxes. Exchange 2003–2016.
  • Staged migration — Batches over time. Exchange 2003/2007 only (legacy).
  • Hybrid migration (minimal or full) — Coexistence between on-prem and online. Supports Exchange 2010+. Recommended for most enterprises. Allows gradual moves with shared calendar/free-busy.
  • Third-party tools — BitTitan MigrationWiz, Quest On Demand Migration, etc. Useful when no hybrid is desired or for cross-forest/IMAP sources.

For organizations with 150+ mailboxes, hybrid is the standard recommendation.

"There are basically four ways to get from Exchange Server to Exchange Online. Cutover is the fastest but caps at 150 mailboxes. Staged is legacy. Third-party tools are great for messy sources. But for any real enterprise, hybrid is the standard — it gives us coexistence, free/busy, and a controlled wave-based move."
What's the difference between Minimal Hybrid and Full Hybrid? Planning
  • Minimal Hybrid (Express) — Simplified wizard, fewer prerequisites. Mailbox migration + basic mail flow. No long-term coexistence. Best for "get out of on-prem fast" scenarios.
  • Full Hybrid — Complete coexistence: shared free/busy, cross-premises mail flow, unified GAL, OAuth-based calendar sharing, MailTips across orgs. Best for phased, enterprise-grade migrations.
"Think of Minimal Hybrid as a one-way door — it's a fast track to move mailboxes and walk away from on-prem. Full Hybrid is what we use when the migration takes months and users on both sides need to share calendars, mail tips, and a single global address list while we move them in waves."

2. Mailbox Lifecycle: On-Prem → Online

End-to-end view of what happens to a single mailbox from the moment it lives on Exchange Server until it is fully cloud-native in Exchange Online. Use this as the operational reference during migration waves.

"Let's walk a single mailbox through the entire journey — from sitting quietly on an Exchange Server in our datacenter, all the way to being fully cloud-native in Exchange Online. Each stage has a clear definition, a clear owner, and a clear exit criterion."
1

On-Prem Native

Mailbox lives in an on-prem database. No cloud footprint. AD account only.

"This is our starting point — every mailbox lives entirely on Exchange Server. No Microsoft 365 dependency yet. Our job here is to take a clean inventory: sizes, rules, delegates, public folder usage."
2

Directory Synced

Entra Connect syncs user to Entra ID as a MailUser. License optional yet.

"Once Entra Connect runs, the user shows up in Entra ID as a MailUser. Nothing changes for the end user — but in the background we now have an identity in the cloud ready to receive a mailbox."
3

Hybrid-Ready

HCW done, Migration Endpoint (MRS) validated, license assigned, batch staged.

"After the Hybrid Configuration Wizard, on-prem and Exchange Online speak the same language. Free/busy works cross-premises, the migration endpoint is healthy, licenses are assigned — the mailbox is now eligible to be moved."
4

Move In Progress

MRS sync of mailbox content to EXO. User keeps using on-prem mailbox.

"This is the silent phase. The Mailbox Replication Service copies content to the cloud in the background — sometimes for hours or days. The user keeps working on-prem, completely unaware. We monitor with Get-MoveRequestStatistics."
5

Cutover (Finalize)

Final incremental sync, mailbox flipped to EXO, on-prem becomes RemoteMailbox.

"This is the only moment the user really feels. Outlook prompts to restart, the profile re-points to Exchange Online, and the on-prem object becomes a stub. We schedule this off-hours so the OST can rebuild without business impact."
6

Online Native

Outlook autodiscovers EXO. Mail flow via EOP. On-prem holds stub only.

"The mailbox is now native in Exchange Online. The user gets bigger quotas, the in-place archive, and modern features. Mail flows through Exchange Online Protection. The on-prem stub stays only for directory consistency."
7

Post-Migration

Archive enabled, retention/MRM applied, hybrid decommissioned when last batch done.

"Once the last batch is done, we flip the MX, retire the hybrid connectors, and apply retention, archive, and Purview labels. We keep one Exchange server for attribute management — or move fully to cloud-managed identity."

What changes for the user at cutover

  • Outlook restarts and recreates the OST against the EXO endpoint (re-download can take hours for large mailboxes — schedule wave on Friday evening when possible).
  • Mobile devices (iOS/Android) usually re-authenticate automatically via Autodiscover; ActiveSync/Modern Auth must be enabled in EXO before cutover.
  • Shared mailboxes and delegates should be migrated in the same batch as their owners to preserve permissions and avoid cross-premises delegation gaps.
  • PST imports and personal archives are activated after the move, not during.
"Cutover is the only moment the user actually feels the migration. Outlook restarts, the OST rebuilds, mobile devices re-authenticate. That's why we batch shared mailboxes with their owners and schedule big mailboxes for Friday evening — so Monday morning everyone just opens Outlook and keeps working."
Watch out: Until the MX record is cut to EXO, mail still hits on-prem first and is forwarded to the cloud mailbox. Keep on-prem transport healthy until the last batch closes.
"Be very careful here — the MX cut is not the first step, it's almost the last. Until then, every inbound email still lands on-prem and gets forwarded to the cloud. If on-prem transport falls over mid-migration, the whole company stops receiving mail."
Identity rule: The on-prem object remains the source of truth for the user (proxyAddresses, displayName, group membership) as long as Entra Connect is in place. All recipient edits stay in on-prem EAC/PowerShell until you fully decommission or enable cloud-managed identity.
"Remember the golden rule of hybrid: as long as Entra Connect is syncing, on-prem wins. Any change made directly in the cloud will be overwritten on the next sync cycle. All mailbox property edits must happen on-prem until we fully decommission."
Done state: A mailbox is "fully migrated" when (a) UserMailbox exists only in EXO, (b) on-prem holds a RemoteMailbox stub or nothing, (c) MX points to EXO, and (d) Outlook profile, mobile, and delegates all operate against EXO without on-prem dependency.
"This is our exit criterion. We don't call a mailbox 'done' just because it moved — we call it done when all four conditions are true. That's the checklist we hand off to operations to close the ticket."

3. Technical Prerequisites

What do we need to set up before starting a hybrid migration? Technical
  • Entra Connect (or Cloud Sync) syncing all migrating users.
  • Exchange Server build at supported CU level for HCW.
  • Public certificate covering autodiscover/owa/mail endpoints.
  • Inbound TCP 443 from EXO to on-prem hybrid endpoint (for MRS proxy).
  • Verified custom domain in Microsoft 365 + MX/SPF/DKIM/DMARC plan.
  • Exchange Online licenses assigned to target users.
  • Hybrid Configuration Wizard run successfully and OAuth configured.
"Before we even think about moving a single mailbox, this checklist has to be green. Identity sync, healthy Exchange build, valid certificate, firewall open for MRS proxy on 443, domain verified, licenses assigned, and HCW completed with OAuth. Skip any one of these and the first batch will fail in a way that's painful to debug."
