Privileged Access · Microsoft Security

PAW — Privileged Access Workstation

A hardened, dedicated device used only for sensitive administrative tasks. Built on the Microsoft Clean Source Principle and the Enterprise Access Model. Complements (does not replace) PIM and Conditional Access.

Audience: Identity / Security / EUC architects · Tiers 0 & 1 administrators

Sections

1. What is a PAW?

A PAW is a workstation that is physically or logically separated from a user's productivity device and is used exclusively to administer high-value systems (Tier 0 / Tier 1). The device itself becomes a security boundary.

Why it exists

Most identity breaches start on a compromised user endpoint (phishing, malicious browser, infected USB). If a Global Admin uses the same laptop as their mailbox and Chrome, one bad click hands the attacker the keys to the kingdom.

Clean Source Principle

A privileged credential can only be used from a device whose trust level is equal to or higher than the resource it administers. A Tier 0 admin must therefore log on from a Tier 0 device.

What a PAW is NOT

  • It is not just a laptop with BitLocker.
  • It is not a jump server (those still get RDP'd into from a dirty endpoint).
  • It is not a replacement for MFA, PIM, or Conditional Access — it works with them.
The core risk PAW solves: credential theft from a productivity endpoint. Without PAW, every Global Admin's daily laptop is effectively a Tier 0 asset — and most are not protected as such.

2. Enterprise Access Model & Tiering

Microsoft replaced the legacy AD "Red Forest / ESAE" tier model with the Enterprise Access Model, which extends tiering to cloud (Entra ID), data, and apps.

3. Tier 0 / 1 / 2 — Deep Dive

The three device profiles are not "more or less secure" — each one protects a different class of identity with different hardware, software, and network rules.

Tier 0 Privileged (true PAW)

Protects: identities that control the identity system itself. Compromise = attacker owns the tenant / forest.

Typical identities:

  • Entra ID: Global Administrator, Privileged Role Administrator, Privileged Authentication Administrator, Partner Tier-2 Support
  • On-prem AD: Domain Admins, Enterprise Admins, Schema Admins, Account Operators
  • Identity infra: AD FS service accounts, AD Connect / Entra Cloud Sync, ADCS (CAs), KDC, DC computer accounts
  • Key Vaults that store identity keys / signing certs

How to decide the tier of an identity

Ask one question: "if this account is compromised, what is the blast radius?"

Can it create/grant permissions to other privileged accounts?  → TIER 0
Can it read/write identity credentials or signing keys?        → TIER 0
Can it administer a production workload (VM/SQL/Storage)?      → TIER 1
Can it deploy to production?                                   → TIER 1
Is it an executive / high-value spear-phishing target?         → TIER 1
Is it a standard user with no elevated privilege?              → TIER 2
Containment rule: a Tier N identity can only be administered by a Tier N (or higher) identity, from a Tier N (or higher) device. Never the inverse.
  • ✅ T0 admin resets a T1 admin's password → OK
  • ❌ T1 admin resets a T0 admin's password → forbidden (T1 becomes a path to T0)
  • ❌ Helpdesk (T2) resets MFA on a Global Admin → forbidden (classic escalation)

4. PAW vs PIM — Complementary, Not Competing

You already understand PIM. The simplest way to frame PAW alongside it:

QuestionPIM answersPAW answers
When can the identity be privileged?Just-in-time, time-bound activation—
For how long?Configurable max duration + approval—
From where can the privilege be used?—Only from a trusted, hardened device
On what device class?—Compliant + filter-tagged as PAW
With what authenticator?Can require MFA on activationEnforces phishing-resistant (FIDO2 / WHfB) at sign-in
 
   ┌──────────────┐    PIM activation     ┌──────────────────────┐
   │  Admin user  │ ───── (JIT + MFA) ──► │  Eligible → Active   │
   └──────┬───────┘                       │   Role assignment    │
          │                               └──────────┬───────────┘
          │ must originate from                      │
          ▼                                          ▼
   ┌──────────────┐  Conditional Access   ┌──────────────────────┐
   │     PAW      │ ─── device filter ──► │  Resource (Tier 0)   │
   │ (clean src)  │      + FIDO2 MFA      │  Entra / AD / Azure  │
   └──────────────┘                       └──────────────────────┘
Golden rule: PIM without PAW = a powerful credential being typed on a malware-friendly device. PAW without PIM = standing privilege that doesn't expire. You want both.

5. Technical Controls (Baseline)

Minimum hardening for a Tier 0 PAW. Anything weaker is a "secure workstation", not a PAW.

Hardware & OS

Must Windows 11 Enterprise

Latest cumulative updates. Autopatch-enrolled. No Home/Pro.

Must TPM 2.0 + Secure Boot + UEFI

Modern Standby capable. Secured-core PC strongly preferred.

Must BitLocker (XTS-AES 256)

Pre-boot PIN for Tier 0. Recovery keys escrowed to Entra ID.

Must VBS + HVCI + Credential Guard

Virtualization-based security. Blocks pass-the-hash / pass-the-ticket.

Identity & Sign-in

Must FIDO2 security key or WHfB

Phishing-resistant MFA. No SMS, no voice, no Authenticator push for Tier 0.

Must No local admin for the user

Separate break-glass local admin, vaulted password, LAPS-managed.

Must Conditional Access — device filter

Privileged roles only sign in from devices tagged extensionAttribute = PAW.

Application & Network

Must WDAC or AppLocker allowlist

Only signed, approved binaries run. Default-deny.

Must No Office / mail / Teams / general browsing

Edge with allowlist of admin portals only.

Must Defender for Endpoint (EDR)

Plan 2, ASR rules in Block mode, tamper protection on.

Recommended Outbound URL allowlist

portal.azure.com, admin.microsoft.com, entra.microsoft.com, intune, etc. Block all else.

Recommended No removable media

USB storage disabled via policy; print disabled for Tier 0.

Optional Always-On VPN / Private Access

For accessing on-prem Tier 0 systems.

6. Implementation Architectures

Pick one based on user count, budget, and operational maturity.

PatternHowProsConsBest for
A. Dedicated hardware PAW Physical laptop, AutoPilot, Intune baseline, FIDO2 key Strongest isolation, no shared kernel with productivity Capex, logistics, admin carries 2 laptops Tier 0 admins, regulated industries
B. Cloud PAW (AVD) Azure Virtual Desktop hardened host pool, accessed from a compliant device Centrally managed, no hardware shipping, fast onboarding Requires strong endpoint posture; network dependency Distributed teams, MSPs
C. Windows 365 Cloud PC as PAW Dedicated Cloud PC SKU + Intune baseline + Zero Trust filter Per-user provisioning, simpler than AVD, Intune-native Cost per user/month; still needs clean source endpoint Modern hybrid orgs, Tier 1 admins
D. PAW + User VM (legacy) Host = PAW, Hyper-V guest = productivity Single device for admin Microsoft no longer recommends — host attack surface is high Avoid for new deployments
Modern default: A = Tier 0 admins, C (Cloud PC) = Tier 1 / specialized. Pattern D should be retired.

7. Implementation Plan (PAW + PIM Integrated)

A phased rollout that gets a small Tier 0 population protected fast, then expands.

Phase 0 — Discovery (Week 1–2)

  • Enumerate all Tier 0 identities: Global Admin, Privileged Role Admin, Privileged Auth Admin, AD DS Domain/Enterprise Admins, AAD Connect / Entra Cloud Sync accounts, ADCS, AD FS service accounts.
  • Map current sign-in devices for those identities (Entra sign-in logs).
  • Inventory existing MFA methods — flag any non-phishing-resistant on Tier 0.
  • Define naming: PAW-T0-*, PAW-T1-*, dynamic Entra groups by device extension attribute.

Phase 1 — Foundation (Week 3–5)

  • Create Intune Security Baseline profile for PAW (Defender, BitLocker, Credential Guard, ASR in Block).
  • Build AutoPilot profile (self-deploy, no user, no local admin).
  • Deploy WDAC policy in Audit → then Enforce.
  • Configure FIDO2 security key sign-in in Entra (Authentication methods policy).
  • Procure / assign FIDO2 keys (2 per admin: primary + backup).

Phase 2 — Conditional Access & PIM (Week 5–6)

  • Create CA policy "CA-T0-PrivilegedRoles-RequirePAW":
    • Assignments: Directory roles = all privileged roles
    • Conditions: Device filter = device.extensionAttribute1 -eq "PAW-T0"
    • Grant: Require compliant device + phishing-resistant MFA + Hybrid join (if applicable)
  • Configure PIM for the same roles: eligible-only, max 4h activation, justification + approval for Global Admin, ticket integration optional.
  • Define 2 break-glass accounts excluded from CA (FIDO2 only, vaulted, monitored).

Phase 3 — Pilot (Week 6–8)

  • Onboard 3–5 Tier 0 admins to physical PAW hardware.
  • Validate admin journeys: portal.azure.com, AD DS RSAT, Exchange Online, Intune, GitHub Enterprise admin, etc.
  • Test break-glass procedure end-to-end (and rotate keys after).
  • Capture friction points; tune URL allowlist and WDAC exceptions.

Phase 4 — Rollout (Week 9–14)

  • Expand to remaining Tier 0 population.
  • Begin Tier 1 rollout using Windows 365 Cloud PC + lighter baseline.
  • Enforce: CA policy now blocks privileged sign-in from non-PAW devices.
  • Decommission standing admin rights on productivity laptops.

Phase 5 — Operate & Monitor (ongoing)

  • Sentinel / Defender XDR rules: privileged sign-in from non-PAW, FIDO2 registration changes, PIM activation anomalies, WDAC block events.
  • Quarterly access review of Tier 0 eligibility (via PIM).
  • Semi-annual PAW image rebuild + WDAC policy refresh.
  • Tabletop exercise: lost FIDO2 key, lost PAW laptop, break-glass invocation.
Quick win sequence (if you only have 30 days): Tier 0 inventory → FIDO2 keys → Intune-managed dedicated laptops → CA device filter → PIM JIT on Global Admin. Everything else is iteration.

8. Operations & Common Pitfalls

Operational must-haves

Break-glass

2 cloud-only Global Admin accounts, FIDO2, excluded from CA, sealed credentials in a physical safe. Sign-in alerts to SOC.

Backup FIDO2 key

Every admin gets 2 keys. Loss of single key must not block emergency access.

PAW lifecycle

Wipe + rebuild on role change. Never reassign a PAW between users without full reimage.

Monitoring

Defender XDR + Sentinel workbook for "privileged sign-in from non-compliant or non-PAW device" — should be 0.

Top pitfalls

PitfallWhy it breaks PAWFix
Admins RDP from their productivity laptop into the "PAW"Clean source violated — credentials enter on a dirty endpointBlock RDP-in; require local sign-in or AVD client from a compliant device
Allowing Office / Teams "just for convenience"Reintroduces phishing & macro attack surfaceHold the line. Use a second device for chat.
Sharing FIDO2 keysDestroys non-repudiation1 key per identity; serial-number tracked
No CA device filterPAW exists but isn't enforced — admins still sign in from anywhereEnforce device filter on all privileged roles
Forgetting AAD Connect / sync accountsHybrid identity sync accounts are Tier 0 — often missedAdd them to PAW-required CA scope
Hyper-V host = productivity, guest = PAWHost compromise = PAW compromiseInvert it (host = PAW) or use separate hardware
Anti-pattern: "We'll just give admins a second login on the same laptop." Same kernel, same browser cache, same attack surface. That is not a PAW.

9. End-to-End Flow — Account, PIM, PAW, Target

A common admin question: "I have a PIM-eligible account — do I just elevate and connect to a machine on the network?" Yes, but the device you elevate from matters as much as the role itself.

 [You + FIDO2 key] ──► sign in to PAW ──► activate PIM ──► access target
     identity            clean device       time-bound        Tier 0 system
                                            elevation

Step by step

1. The account

  • One dedicated admin identity (e.g. adm-ofernando@tenant), separate from your daily user account.
  • It is PIM-eligible for a role — assigned but not active. Zero standing privilege.

2. The device (PAW)

  • Sign in to the PAW with the admin account + FIDO2 key.
  • PAW is Intune-compliant and tagged (e.g. extensionAttribute1 = PAW-T0).
  • Conditional Access enforces: privileged roles usable only from this device class.

3. The elevation (PIM activation)

  • From the PAW, open Entra → PIM → Activate the role.
  • Provide justification, optional ticket, optional approval.
  • Role becomes active for a limited window (e.g. 4 h).

4. The access

  • With the role active AND signed in from PAW, reach the target:
  • Azure portal, Entra admin, Intune, Exchange admin
  • RDP/SSH via Azure Bastion or a hardened jump host into a restricted-network server
  • Admin PowerShell (Graph, Az, ExchangeOnline) from the PAW itself

5. Timer expires

  • Role auto-deactivates. You're back to a standard user.
  • Sign-in logs + PIM audit log record every action.

Two valid target-access models

ModelShapeWhen to use
A. PAW as the admin endpoint Admin tools (browser, PowerShell, RSAT, Bastion client) run directly on the PAW. PAW has network paths (VPN / Private Access) to reach management endpoints. Cloud-first admin (Entra, Azure, M365). Preferred modern model.
B. PAW → jump host → target From the PAW, open RDP/SSH (via Azure Bastion or hardened jump server) into a management VM inside the restricted network; from there reach Tier 0 systems. On-prem AD DS, legacy systems, segmented datacenter networks.
Critical: the jump host is not a substitute for the PAW. RDP/SSH must be opened from the PAW to the jump host — never from a daily laptop. Otherwise the credential touches a dirty device and Clean Source is broken.

Minimum ingredients checklist

  • Separate admin account (adm-*), cloud-only when possible
  • FIDO2 security key (×2 — primary + backup)
  • PIM-eligible assignment (not active / not permanent)
  • PAW device — Intune-managed, tagged, compliant
  • Conditional Access policy: privileged roles → require PAW device filter + phishing-resistant MFA
  • (If on-prem targets) Azure Bastion or hardened jump host in the management subnet
  • 2 break-glass Global Admin accounts excluded from CA, vaulted, monitored

10. Direction of Access — You go TO the PAW

A frequent point of confusion: "Do I access the PAW to elevate, or do I use a jump host to access the PAW?" The PAW is the destination, not a stop along the way.

Correct direction: 
[You + FIDO2] ──sign in──► [PAW] ──activate PIM──► [Target / Bastion / Jump host]
The PAW is where the admin credential lives and is typed. Nothing should sit in front of the PAW between you and the keyboard, because that "something in front" would be a dirty device handling your credential — Clean Source violated.
Wrong pattern — jump host → PAW: 
[Daily laptop] ──RDP──► [Jump host] ──RDP──► [PAW]   ✗ BROKEN
Your admin password / FIDO2 PIN / session token would be processed by the daily laptop's keyboard, clipboard, and screen-capture surface. The PAW becomes pointless.

The two valid access shapes

Shape 1 — PAW is the physical thing in front of you

You ──► PAW (laptop on your desk) ──► Azure portal / Bastion / admin tools

You literally sit at the PAW. FIDO2 key plugged into the PAW. Best for Tier 0.

Shape 2 — Cloud PAW (Windows 365 / AVD)

You ──► Compliant endpoint ──► PAW Cloud PC ──► Azure portal / Bastion / admin tools
        (Intune-managed,           (the real
         hardened, no admin)        "clean" device)

The "compliant endpoint" used to launch the Cloud PC must itself be hardened and Intune-compliant — not a random laptop. The Cloud PC is the PAW; the endpoint is just a screen/keyboard.

Where the jump host fits

The jump host (or Azure Bastion) comes after the PAW, not before:

You ──► PAW ──► PIM activate ──► Bastion / jump host ──► Tier 0 server
                                  (only reachable
                                   from PAW)

Use a jump host when the target lives on a restricted management network that the PAW can't reach directly. The jump host's RDP/SSH session is initiated from the PAW.

TL;DR

QuestionAnswer
Do I go to the PAW first?Yes. PAW is the trusted device where you sign in and activate PIM.
Do I use a jump host to reach the PAW?No. That defeats the purpose.
When do I use a jump host?After the PAW, to reach segregated networks.
Where do I plug the FIDO2 key?Into the PAW (or the endpoint launching the Cloud PC PAW).

11. Q&A — Common Client Questions

Do we really need separate hardware? Can't a VM be enough?

For Tier 0, separate hardware (or a properly isolated Cloud PC accessed from a compliant device) is the defensible answer. A productivity-host VM-guest split puts the host kernel — the most exposed surface — in the trust path of your most privileged identity.

How does PAW interact with Zero Trust?

PAW is the device-identity pillar of Zero Trust for privileged scenarios. Conditional Access verifies user + device + signal; PAW guarantees the device signal is trustworthy enough for the resource being accessed.

What about MacOS / Linux admins?

Microsoft's PAW guidance is Windows-centric because the controls (Credential Guard, WDAC, HVCI) are Windows features. For non-Windows admins, the equivalent is a dedicated, hardened, Intune/Jamf-managed device with the same isolation principles — but tooling parity is weaker.

Cost — is this viable for a 5,000-seat org?

You don't PAW everyone. A typical org has 10–50 Tier 0 identities. That's 10–50 dedicated laptops or Cloud PCs — small spend compared to the cost of a Global Admin compromise. Tier 1 (a few hundred) usually goes to Cloud PC.

Does PAW remove the need for PIM?

No. PAW controls the device; PIM controls the time and scope of the privilege. Use both. Together with FIDO2 and Conditional Access they form the "Securing Privileged Access" roadmap.

How do admins do their regular email / Teams?

From their normal Enterprise-tier device, with a separate, non-privileged user account. Admin identity stays exclusively on the PAW. Two accounts, two devices, one human.

What if the FIDO2 key is lost?

Use the backup key. If both are lost, the admin loses access until re-enrolled in person (or via verified video) by the identity team. Break-glass accounts cover catastrophic scenarios.
