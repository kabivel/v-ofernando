FIDO2 + Certificate (Intune) at the Same Time

How to require two phishing-resistant credentials together — a FIDO2 security key (the user) plus a certificate that Microsoft Intune delivers to the device (device trust). The honest truth about what Entra supports, plus the exact configuration that actually works.

Audience: Identity / Security / EUC architects · High-assurance and Tier-0 admin scenarios

Jump to a section

1 · The Concept

You want a sign-in that demands two independent strong credentials together: a FIDO2 security key (the user) and proof that the session runs on a managed device trusted via the certificate Intune deployed (the device). This is "defense in depth" applied to authentication.

🔑 FIDO2 Security Key

A hardware authenticator (e.g. YubiKey, Feitian) using WebAuthn. Public/private key pair bound to the Entra account; the private key never leaves the device. Unlocked by PIN or biometric.

Phishing-resistant MFA on its own

📜 Certificate via Intune

An X.509 certificate provisioned by Intune (device registration for compliance/hybrid join, or a SCEP/PKCS user cert). In this design it primarily anchors device trust so Conditional Access can require a managed device.

Phishing-resistant Device or user

🎯 The Goal

Force both together, not "either/or". The certificate proves the device; the FIDO2 key proves the user. Two grant controls ANDed in Conditional Access = two credentials in one access decision.

High assurance

Key reality first: Entra cannot require two user credentials (FIDO2 and a user certificate) together inside a single authentication strength — not even with a "single-factor certificate" trick. The only AND-combinations Entra offers pair a password / federated sign-in with "something the user has" (SMS, voice, push, OATH) — never FIDO2 or a certificate. The supported way to demand two credentials at once is to AND two Conditional Access grant controls: a phishing-resistant authentication strength (FIDO2 = the user) and a managed/compliant device whose trust is anchored by the Intune-deployed certificate (the device). Section 2 proves this with the docs.

2 · What Entra Actually Allows (the honest part)

Authentication Strengths are built from combinations of methods. Each entry in a strength is a complete, alternative way to sign in — the entries are joined by OR, not AND. The user satisfies the strength with any one allowed combination.

You cannot require FIDO2 + a certificate together in one authentication strength. This is the most common misconception. Neither "two multifactor methods" nor a "single-factor cert + FIDO2" combination exists for this purpose. The Microsoft docs are explicit (see below).

The only AND-combinations Entra offers

Entra does have combinations that require two methods together — but they all follow one shape: a primary single-factor sign-in (password or federated) plus "something the user has". And per Microsoft Learn, "something the user has" is defined precisely as:

"Something the user has refers to one of the following methods: text message, voice, push notification, software OATH token, or hardware OATH token."

FIDO2 and certificate are not on that list. They are never part of an AND-combination — they only ever appear as standalone combinations (joined by OR with everything else).

Combination typeExamplePairs with FIDO2 / cert?
Primary + something you havePassword + push, Federated SF + OATH❌ No
FIDO2 security keyStandalone (MF on its own)— standalone only
Certificate (single- or multifactor)Standalone— standalone only

So how do you require BOTH at once?

You split the two credentials across two Conditional Access grant controls and join them with AND ("Require all the selected controls"):

Grant control 1 — Require authentication strength = Phishing-resistant MFA → forces the FIDO2 security key (the user).
Grant control 2 — Require device to be compliant (or Hybrid Entra joined) → only satisfied by a managed device, whose trust is anchored by the Intune-deployed certificate (the device).
Operator = AND. Result: the user must present a FIDO2 key and be on the certificate-bearing managed device, in the same access decision.
What you wantSupported?How
FIDO2 OR Certificate (either is fine)YesOne custom strength listing both standalone combinations
FIDO2 AND user-certificate in one auth strengthNoNot expressible — Entra has no such combination
FIDO2 (user) AND Intune cert (device trust)YesTwo CA grant controls ANDed — this guide
Why this is the right model: the certificate's natural job is to prove the device (compliance / Entra hybrid join), while FIDO2 proves the user. ANDing them gives genuine two-credential assurance — a stolen FIDO2 key is useless off a managed device, and a managed device is useless without the key.

3 · Deploy the Certificate with Intune

The Intune-managed device is what supplies the device factor. For the supported AND model this comes from Intune enrollment + compliance (and/or hybrid join), not from a hand-built user certificate. A user certificate (SCEP/PKCS) is only needed for the optional CBA sign-in path.

Two roles a certificate can play — know which you need:
  • Device trust (this guide's AND): achieved by Intune enrollment + a compliance policy and/or Microsoft Entra hybrid join. The device registration certificate is provisioned automatically by that process — you do not hand-build a user SCEP cert for this. This is what satisfies the "Require compliant device" control.
  • User sign-in (CBA, optional): a user certificate the person presents at sign-in, built with the SCEP/PKCS profile below and mapped by UPN. This is an OR alternative to FIDO2 — it cannot be ANDed with it (Section 4).
For the two-credential AND, you usually don't need the SCEP user cert at all. Enroll the device in Intune, apply a compliance policy (and/or hybrid join), and the "compliant device" control is satisfied. The SCEP/PKCS profile below is only required if you also want the certificate usable as a user sign-in method (Section 4).

If you need the user certificate (CBA path) — profile options

Prerequisites

  • Enterprise PKI (AD Certificate Services) with a certificate template for Client Authentication EKU (1.3.6.1.5.5.7.3.2).
  • Trusted Root certificate profile in Intune that pushes your Root (and Issuing) CA chain to devices.
  • Intune Certificate Connector installed and healthy.
  • The Subject / SAN must carry an attribute Entra can map to the user (commonly SAN → Principal Name = userPrincipalName, or RFC822Name = email).

Build the SCEP profile (Intune admin center)

  1. Push the CA chain first: Devices → Configuration → Create → Templates → Trusted certificate (one for Root, one for Issuing CA).
  2. Create Devices → Configuration → Create → Templates → SCEP certificate.
  3. Certificate type: User (so it maps to the signing-in identity).
  4. Subject name format: CN={{UserPrincipalName}}.
  5. Subject alternative name: add User principal name (UPN) = {{UserPrincipalName}}. critical for Entra mapping
  6. Key usage: Digital signature + Key encipherment. Key storage provider: Enroll to TPM if present.
  7. Extended key usage: Client Authentication (1.3.6.1.5.5.7.3.2).
  8. Root Certificate: reference the Trusted certificate profile from step 1.
  9. SCEP Server URLs: your NDES endpoint. Save and assign to your pilot group.
Mapping must match Entra. Whatever attribute you place in the SAN (UPN, email/RFC822, or the cert serial/issuer) has to line up with the username binding you configure in Entra CBA (Section 4). A mismatch = "certificate cannot be mapped to a user".

4 · Optional — Certificate as a User Credential (CBA)

This section is only needed if you also want users to be able to sign in with the certificate itself (e.g. as an OR alternative to FIDO2, or for smartcard scenarios). It is not what creates the AND with FIDO2 — that lives in Conditional Access (Section 7).

Do not expect CBA to combine with FIDO2. Whether you set the CBA binding to single-factor or multifactor, Entra treats a presented user certificate as its own standalone combination. There is no authentication-strength combination that requires "certificate AND FIDO2" in one sign-in. If your goal is purely the two-credential AND, you can skip this section and rely on device trust (Section 3) + FIDO2 (Section 5).

Step 1 — Upload your CA chain

  1. Entra admin center → Protection → Show more → Security → Certificate authorities (or the PKI blade under Authentication methods).
  2. Upload Root CA and Issuing CA .cer files; provide the CRL Distribution Point URL for each.
  3. Mark the issuing CA appropriately and save.

Step 2 — Enable & tune CBA

  1. Protection → Authentication methods → Policies → Certificate-based authentication.
  2. Enable = Yes. Target your pilot group.
  3. Open the Configure tab.
  4. Authentication binding → choose single-factor or multifactor protection level per issuer / policy OID, depending on how strong you want a cert-only sign-in to count.
  5. Username binding → choose the field that matches your Intune SAN, e.g. PrincipalName → userPrincipalName.
  6. Save.
You can use certificate issuer / policy OID rules in the authentication binding and in the custom authentication strength advanced options to restrict which certificates qualify. Useful when reserving a specific Intune-issued cert for a high-assurance app.

5 · Enable FIDO2 Security Keys

Turn on FIDO2 in the same Authentication methods policy and let pilot users register a key.

  1. Protection → Authentication methods → Policies → FIDO2 security key.
  2. Enable = Yes. Target = pilot group.
  3. Configure tab → Allow self-service set up = Yes.
  4. Enforce attestation = Yes (only keys with valid vendor attestation).
  5. Enforce key restrictions = optional — add allowed AAGUIDs to permit only approved key models (e.g. corporate-issued YubiKeys).
  6. Save.

User registration

  • Users go to https://aka.ms/mysecurityinfo → Add sign-in method → Security key.
  • First registration needs an existing strong credential. If the user has none, issue a Temporary Access Pass (TAP) from Users → Authentication methods → Add → Temporary Access Pass to bootstrap.
Pair the registration carefully: the same physical key flow should be tested end-to-end against the auth strength before broad rollout.

6 · Create the Phishing-Resistant Authentication Strength

This strength enforces the user factor: FIDO2. The device factor (Intune certificate) is enforced separately in Conditional Access (Section 7). You can use the built-in Phishing-resistant MFA strength or build a custom one scoped to just FIDO2.

  1. Protection → Authentication methods → Authentication strengths → New authentication strength.
  2. Name it e.g. FIDO2 only — phishing-resistant.
  3. Under Phishing-resistant MFA, select Passkeys (FIDO2). Optionally also allow Windows Hello for Business and Certificate-based authentication (multifactor) if you want those as OR alternatives.
  4. (Optional) Open Passkeys (FIDO2) → Advanced options and add allowed AAGUIDs to permit only corporate key models.
  5. Review and create.
There is no "FIDO2 + certificate" combination to select here. Everything you tick in a strength is an alternative (OR). To require a second credential you must add a second grant control in Conditional Access — that is the next section.
If you select…Sign-in is satisfied by…
Only Passkeys (FIDO2)A FIDO2 key (then CA still requires a compliant device — the AND)
FIDO2 + CBA (MF)A FIDO2 key or a multifactor certificate — user's choice
FIDO2 + Windows HelloA FIDO2 key or Windows Hello for Business

7 · Enforce Both Credentials with Conditional Access

This is where "both at once" actually happens. You combine two grant controls with AND: the FIDO2 authentication strength and a managed-device requirement satisfied by the Intune certificate.

  1. Protection → Conditional Access → Create new policy.
  2. Users: your pilot group / Tier-0 admins. Always exclude break-glass accounts.
  3. Target resources: the sensitive app(s), or "All resources" for admin roles.
  4. Grant → Grant access, then select both:
    • ✅ Require authentication strength → your FIDO2 strength (the user).
    • ✅ Require device to be marked as compliant (and/or Require Microsoft Entra hybrid joined device) — satisfied via the Intune-deployed certificate (the device).
  5. Set "Require all the selected controls" (this is the AND).
  6. Start in Report-only, validate sign-in logs, then switch to On.
The result: a user can only get in if they (1) authenticate with a FIDO2 key and (2) are on a managed device carrying the Intune certificate. Two independent credentials, enforced in a single access decision.
Break-glass: always exclude at least two emergency-access accounts from this policy, and verify they can still sign in. A misconfigured AND policy can lock everyone out.
If you truly need the certificate as a presented user credential (not device trust) together with FIDO2 in the same sign-in, Entra cannot do it. Re-evaluate whether device trust (above) meets the requirement — in almost all cases it does, and more securely.

8 · PowerShell / Microsoft Graph

Reference snippets to enable methods, create the strength, and bind it in Conditional Access. Validate in a test tenant first.

Connect

# Microsoft Graph PowerShell SDK
Connect-MgGraph -Scopes "Policy.ReadWrite.AuthenticationMethod",`
  "Policy.ReadWrite.ConditionalAccess",`
  "Policy.Read.All"

Enable FIDO2 for a group

$fido2 = @{
  id    = "Fido2"
  state = "enabled"
  includeTargets = @(@{ targetType="group"; id="<pilotGroupId>" })
  isAttestationEnforced = $true
  isSelfServiceRegistrationAllowed = $true
}
Update-MgPolicyAuthenticationMethodPolicyAuthenticationMethodConfiguration `
  -AuthenticationMethodConfigurationId "Fido2" -BodyParameter $fido2

Enable CBA (optional — only if cert is also a user sign-in method)

# Optional. CBA lets users sign in WITH the certificate (an OR alternative to FIDO2).
# It does NOT create an AND with FIDO2 - that is done in Conditional Access.
$cba = @{
  "@odata.type" = "#microsoft.graph.x509CertificateAuthenticationMethodConfiguration"
  id    = "X509Certificate"
  state = "enabled"
  includeTargets = @(@{ targetType="group"; id="<pilotGroupId>" })
  certificateUserBindings = @(
    @{ x509CertificateField="PrincipalName"; userProperty="userPrincipalName"; priority=1 }
  )
  authenticationModeConfiguration = @{
    x509CertificateAuthenticationDefaultMode = "x509CertificateSingleFactor"
  }
}
Update-MgPolicyAuthenticationMethodPolicyAuthenticationMethodConfiguration `
  -AuthenticationMethodConfigurationId "X509Certificate" -BodyParameter $cba

Create the FIDO2 authentication strength

# allowedCombinations are joined by OR. Each entry is one complete way to sign in.
# There is NO "x509 + fido2" combined token - that does not exist in Entra.
$body = @{
  displayName = "FIDO2 only - phishing-resistant"
  description = "Requires a FIDO2 security key (the user factor)"
  allowedCombinations = @( "fido2" )   # optionally add "windowsHelloForBusiness","x509CertificateMultiFactor"
}
New-MgPolicyAuthenticationStrengthPolicy -BodyParameter $body
Each string in allowedCombinations is a single combination. Comma-separated tokens (e.g. "password,sms") only exist for the documented "primary + something you have" pairs. FIDO2 and certificate have no comma-paired form — confirm with Get-MgPolicyAuthenticationStrengthPolicy on your tenant.

Conditional Access — AND two grant controls

# The "both at once" is here: authenticationStrength (FIDO2) AND compliantDevice,
# joined by operator AND.
grantControls = @{
  operator = "AND"                              # Require ALL selected controls
  builtInControls = @( "compliantDevice" )      # device trust via Intune cert
  authenticationStrength = @{ id = "<fido2StrengthPolicyId>" }
}
operator = "AND" plus both a builtInControls entry and an authenticationStrength is exactly the two-credential requirement: FIDO2 (user) and a managed device carrying the Intune certificate (device).

10 · FAQ & Gotchas

Why can't I just require FIDO2 AND a certificate in one authentication strength?

Because Entra has no such combination. Authentication-strength combinations are joined by OR, and the only AND-combinations are "password / federated single-factor + something the user has" — where "something the user has" is strictly SMS, voice, push, software OATH, or hardware OATH. FIDO2 and certificate are never AND-paired. To require two credentials, AND two Conditional Access grant controls instead.

So how do I get "two credentials at once"?

Use Conditional Access with Require authentication strength (FIDO2) AND Require device to be compliant / hybrid joined, set to "Require all the selected controls". The Intune certificate anchors device trust (the device factor); FIDO2 is the user factor.

Does a single-factor certificate help here?

Not for ANDing with FIDO2 — single-factor vs multifactor only changes how a cert-only sign-in is counted, it never makes the cert combinable with FIDO2 in a strength. The certificate's real value in this design is anchoring device compliance/trust.

"Certificate cannot be mapped to a user." (if using CBA)

The SAN/Subject attribute in the Intune profile doesn't match the Entra username binding. Align them — e.g. SAN UPN {{UserPrincipalName}} in Intune ↔ PrincipalName → userPrincipalName in Entra CBA.

Do I need devices to be Intune-compliant?

Yes — for this design the Require compliant device (or hybrid joined) control is what enforces the second credential. The Intune-deployed certificate is what lets the device prove that trust.

What about break-glass accounts?

Always exclude two emergency-access accounts from this Conditional Access policy and keep their credentials secured offline. Test their sign-in before enforcing.

Can I do this for only one critical app?

Yes. Scope the Conditional Access policy's Target resources to that single app. Other apps keep a lighter policy. Use Sign-in frequency for step-up re-authentication.

Roll-out order recommendation?
  1. Stand up PKI + Intune cert profiles → verify devices show as compliant / hybrid joined.
  2. Enable FIDO2 → register a pilot key.
  3. Build the FIDO2 authentication strength (or use built-in Phishing-resistant MFA).
  4. Create the CA policy ANDing the strength + compliant device.
  5. Run in report-only, inspect logs, then enforce.
