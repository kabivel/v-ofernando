Identity Guideline

Okta, Active Directory, and Microsoft Entra ID

A practical discovery guide for understanding what Okta is doing today, whether AD still feeds Microsoft 365 identities, and whether Entra Connect or Cloud Sync is still required.

Scope: Okta role discovery, Entra Connect decision support, reference patterns, risks, and client conversation support

Navigate by topic

1. Executive Summary

Use this section to align stakeholders quickly before going into technical details.

What Okta does

Okta is an Identity and Access Management platform. It can provide single sign-on, adaptive MFA, sign-on policies, application assignments, lifecycle automation, and provisioning into downstream applications.

How it fits with AD

In many enterprise environments, Active Directory remains the on-premises directory and source for users, groups, passwords, and domain access. Okta commonly integrates with AD through the Okta AD Agent.

How it fits with Entra ID

Microsoft Entra ID remains required for Microsoft 365, Azure, Teams, Exchange Online, SharePoint, Intune, and Microsoft security workloads. Even if Okta handles sign-in, Entra ID still holds the Microsoft cloud identity.

Key idea: Because the customer already has Okta, the first question is not whether Okta exists. The first question is what Okta owns today: sign-in only, MFA only, app assignment, provisioning, lifecycle, or the full identity source model.
Decision guardrail: Do not remove Entra Connect just because Okta is present. If AD is still the authoritative source for Microsoft 365 identities, Entra Connect or Cloud Sync is usually still required.

Sync

Use this section as a preparation view. It explains how Okta integrations usually work internally, without going into implementation or troubleshooting steps.

Core idea: Okta is not just a login page. It is an identity control plane that can import identities, normalize profiles, evaluate policy, broker authentication, assign applications, provision accounts, and emit audit events.

Main building blocks

Okta org

The tenant boundary for the customer. It contains users, groups, applications, policies, directories, integrations, logs, and administrative configuration.

Universal Directory

Okta's user and group profile store. It holds normalized identity attributes and can map attributes between sources such as AD, HR, Entra ID, and SaaS applications.

Applications

Application objects define how users access a target system. An app can have SSO settings, provisioning settings, group assignments, attribute mappings, and sign-on policy behavior.

Groups

Groups are commonly used to assign apps, drive provisioning, apply policies, and organize entitlements. Groups can come from AD, Okta, HR, or other systems.

Policies

Policies decide how access is evaluated. They can include password rules, sign-on rules, MFA requirements, device posture signals, network zones, risk, and application context.

System Log

Okta records authentication, provisioning, lifecycle, admin, policy, and integration events. For discovery, logs help reveal what Okta is actually doing, not just what people think it is doing.

Inbound identity flows

Source How it commonly connects What Okta may receive
Active Directory Okta AD Agent installed on Windows servers in the customer network. Users, groups, selected attributes, account status, and sometimes delegated password authentication or password operations.
LDAP directory Okta LDAP Agent. Users, groups, and attributes from a non-AD LDAP source.
HR system Prebuilt connector, API integration, import, or lifecycle management workflow. Worker status, start date, termination date, department, manager, title, location, and employee identifiers.
Admin or API Manual creation, bulk import, Okta API, or automation platform. Cloud-native Okta users, service identities, test users, contractors, or non-employee accounts.

Authentication and SSO flows

SAML

Common for enterprise SaaS. Okta authenticates the user and sends a signed SAML assertion to the application. The app trusts Okta as the Identity Provider.

OIDC / OAuth

Common for modern web and mobile apps. Okta issues tokens through standards-based flows so the application can validate identity and access claims.

WS-Fed / federation to Microsoft

In Microsoft 365 federation scenarios, Entra ID can redirect authentication to Okta for federated domains. Entra ID still remains the Microsoft cloud directory.

Delegated authentication

When integrated with AD, Okta may validate passwords against AD instead of storing Okta-managed credentials. This keeps AD in the authentication path.

Provisioning and lifecycle flows

Flow What happens Why it matters
Assignment A user or group is assigned to an application in Okta. This can trigger app visibility, SSO entitlement, provisioning, or license assignment depending on the integration.
Provisioning Okta creates or updates an account in the target application through SCIM, API, or a connector. This is how Okta can automate joiner and mover processes for many SaaS applications.
Attribute mapping Okta maps source attributes to target app attributes. Attribute ownership matters. A value may be mastered by HR, AD, Okta, or the application itself.
Deactivation When a user is deactivated or unassigned, Okta may suspend, deactivate, or remove the downstream account. This is one of the most important controls for leaver processes and security risk reduction.
Writeback Okta writes a supported change back to a connected system, such as AD or an application. Writeback is not universal. It depends on the target system, connector capability, and configured ownership model.

Microsoft-specific view

Okta as IdP for Microsoft 365

Okta may handle sign-in and MFA for Microsoft 365 through federation. This does not automatically mean Okta creates or updates Entra ID users.

Okta provisioning to Microsoft cloud

Okta may provision users or attributes into Microsoft cloud services in some scenarios. This must be distinguished from AD-to-Entra synchronization by Entra Connect or Cloud Sync.

Entra ID remains the Microsoft authority

For Microsoft 365 and Azure, Entra ID still stores the Microsoft cloud identity, licenses, groups, roles, Conditional Access scope, and audit context.

Preparation takeaway: In discovery, separate four questions: where the user is mastered, who authenticates the user, who provisions the user, and who removes access. Okta may own one, several, or all of these, depending on the customer's configuration.

5. Integration Patterns to Identify

Who is the source of truth?

The source of truth is the system considered authoritative for identity data. It may be Active Directory, Okta, an HR platform, or an identity governance system. Avoid assuming that the system used for login is also the source of identity data.

Who authenticates Microsoft 365 users?

Entra ID may authenticate users directly, or it may redirect authentication to Okta through federation. Confirm the domain authentication type, federation settings, and whether staged rollout or cloud authentication is in use.

Who enforces MFA?

MFA can be enforced in Okta, Microsoft Entra ID, or both. Duplicated controls can create repeated prompts, inconsistent access decisions, and complex incident troubleshooting.

Who provisions and disables users?

Provisioning determines how users are created, updated, assigned, suspended, and removed from target systems. The leaver process is especially important because it controls how quickly access is removed after termination.

Where are groups mastered?

Groups may originate in AD, Okta, Entra ID, or individual applications. Identify which groups are used for app assignment, licensing, Conditional Access, privileged access, and operational distribution lists.

6. Can Okta Fully Replace On-Premises AD?

Short answer: No, not fully, if the organization still depends on traditional Active Directory capabilities.

Can overlap Identity and SaaS access

Okta can reduce dependency on AD for SSO, MFA, application access, lifecycle management, provisioning, and user access to cloud or SaaS applications.

Cannot fully replace Windows domain services

Okta does not fully replace classic AD functions such as domain join, Kerberos, NTLM, Group Policy, LDAP for legacy applications, file server ACLs, domain trusts, and domain-based server administration.

When replacement becomes realistic

A full AD retirement usually requires a broader cloud-first program: modern device management, Entra ID, Intune, SaaS migration, application modernization, and removal of legacy LDAP/Kerberos dependencies.

Okta can replace many SaaS identity use cases.
Okta does not fully replace AD for Windows domain, Kerberos, NTLM, GPO, LDAP, and legacy infrastructure.

7. Can Okta Be the Source of Truth if AD Is No Longer Needed?

Short answer: Yes, Okta can become the primary identity source and lifecycle orchestration layer, if the customer has removed on-premises AD dependencies and all required systems can be provisioned and governed from Okta or an upstream HR platform.

Possible Okta as primary identity source

If AD is no longer required, Okta can hold users, groups, app assignments, sign-on policies, MFA enrollment, and lifecycle rules. It can become the primary place where identity and access are orchestrated.

Better pattern HR as employee source

For employees, the HR system is often the best authoritative source for worker status, start date, department, manager, and termination events. Okta then orchestrates identity lifecycle and downstream access.

Microsoft still exists Entra ID remains required

Even without AD, Microsoft Entra ID still exists for Microsoft 365, Azure, Teams, Exchange Online, SharePoint, Intune, and Microsoft security workloads. Okta can feed or federate with Entra ID, but Entra ID does not disappear.

Required conditions

Condition Why it matters
No AD-dependent workloads remain There should be no remaining dependency on domain join, Kerberos, NTLM, GPO, LDAP, AD file permissions, or legacy line-of-business applications tied to AD.
Lifecycle ownership is clear Joiner, mover, and leaver processes must clearly define whether HR, Okta, Entra ID, or another platform owns each action and attribute.
Downstream provisioning is complete Microsoft 365, SaaS applications, security tools, and business systems must receive the right users, attributes, groups, roles, and deactivation events.
Administrative and emergency access is designed Break-glass accounts, privileged roles, recovery procedures, and federation bypass scenarios must be documented and tested. 
Recommended mature model:

HR system = source of truth for employee status
Okta = identity lifecycle and access orchestration
Entra ID = Microsoft cloud identity
Intune = device and application management
Client-ready wording: If the customer no longer needs on-premises AD, Okta can become the primary identity source and lifecycle orchestration layer, provided that all required downstream systems can be provisioned and governed from Okta. For employees, HR is often the best source of truth, with Okta orchestrating identity and access.

8. Does This Customer Still Need Entra Connect?

Short answer: it depends on what Okta is doing today. If Okta is only the Identity Provider for SSO/MFA and AD still creates Microsoft 365 identities, Entra Connect or Cloud Sync is usually still needed.

Unknown scenario Start with role discovery

The customer already has Okta, but that does not tell us whether Okta is only handling authentication or also owning provisioning and lifecycle into Entra ID.

Do not assume Okta replaces sync

Okta and Entra Connect solve different problems. Okta commonly handles sign-in, MFA, SSO, and app access. Entra Connect synchronizes AD identities into Entra ID.

Possible removal Only after source model changes

Entra Connect may not be needed if users are cloud-only in Entra ID and Okta or HR-to-Okta provisioning fully owns user creation, updates, deactivation, and access lifecycle.

AD is still the source of truth for users Entra Connect or Cloud Sync likely still needed Microsoft 365 identity still depends on the on-premises directory source.
Entra ID users show as synchronized from on-premises Sync still exists in the identity model Removing the sync engine without changing the identity model can break lifecycle management.
Exchange Hybrid, hybrid device join, password writeback, or group writeback exists Validate carefully These dependencies often require Microsoft sync capabilities or a planned replacement design.
HR or Okta provisions cloud-only users directly to Entra ID Entra Connect may not be needed This can work if all required attributes, licenses, groups, deactivation flows, and admin recovery paths are covered.

If Okta is the LDAP master

Important distinction: Okta being the LDAP master does not automatically remove the need for Entra Connect. Entra Connect is still relevant when Entra ID depends on AD DS-sourced users, attributes, passwords, groups, devices, Exchange objects, or writeback flows.
Okta as LDAP master = Okta owns the directory/source side for some or all identities
Entra Connect = Microsoft AD DS to Entra ID synchronization engine

Key question:
Does Microsoft 365 still need anything synchronized from AD DS into Entra ID?
Function or dependency Why Entra Connect / Cloud Sync may still be necessary
AD DS-sourced users If users in Entra ID are still synchronized from on-premises AD DS, the Microsoft identity lifecycle still depends on a Microsoft sync engine.
Exchange Hybrid and Exchange attributes Hybrid environments may depend on AD-mastered mail attributes such as proxyAddresses, mail, mailNickname, targetAddress, and msExch* attributes.
Password Hash Sync If Microsoft cloud authentication uses password hash sync from AD DS, Entra Connect is the component that synchronizes password hashes to Entra ID.
Pass-through Authentication If pass-through authentication is used, Microsoft agents validate sign-ins against AD DS. This keeps AD DS in the Microsoft authentication path.
Password writeback / SSPR writeback If users reset passwords in the cloud and those changes must be written back to AD DS, Microsoft sync/writeback capability may still be required.
Group writeback If cloud groups need to be represented back in AD DS, this is a Microsoft sync/writeback scenario rather than a generic LDAP-master scenario.
Device writeback or hybrid device dependencies Some hybrid device designs rely on AD DS and Microsoft sync behavior. Validate before removing Entra Connect or Cloud Sync.
Hybrid Microsoft Entra Join If Windows devices are hybrid joined, there is usually an AD DS plus Entra ID relationship that must be understood before changing sync.
AD DS-sourced groups or application dependencies Groups may still originate in AD DS and be used for Microsoft 365 licensing, access, mail distribution, or application authorization.
Microsoft-supported hybrid identity model Some Microsoft hybrid scenarios expect Entra Connect or Cloud Sync as the supported AD DS to Entra ID synchronization path.
Client-ready wording: Even if Okta is the LDAP master, are Microsoft 365 users, attributes, passwords, groups, devices, or Exchange objects still synchronized into Entra ID from AD DS through Entra Connect or Cloud Sync? If yes, there is still a Microsoft sync dependency.

9. Can Okta Fully Replace Entra Connect?

Short answer: Usually no as a direct replacement, especially in classic Microsoft hybrid identity.

Can overlap Provisioning outcomes

Okta can provision users into Microsoft 365 or Entra ID in some scenarios through supported integrations, APIs, and lifecycle workflows.

Not equivalent Hybrid identity engine

Entra Connect is the Microsoft synchronization engine for AD-to-Entra hybrid identity. It handles many Microsoft-specific object, attribute, password, and hybrid workload scenarios.

Common reasons Entra Connect remains

  • AD remains the source of authority.
  • Users are synchronized or hybrid.
  • Exchange hybrid attributes are required.
  • Password hash sync or pass-through authentication is used.
  • Hybrid join, writeback, or Microsoft-supported hybrid state is required.
Okta can provision users into Entra ID in some models.
Okta is not a full drop-in replacement for Entra Connect in classic Microsoft hybrid identity.
The key discovery question is whether Okta is only IdP/SSO or also provisioning/lifecycle authority.

10. Can Okta Fully Replace Cloud Sync?

Short answer: Not as a direct replacement for Microsoft's AD-to-Entra synchronization model.

Can overlap Some provisioning flows

Okta may import users from AD and provision accounts to downstream systems, including Microsoft cloud targets in supported scenarios.

Different purpose Microsoft sync service

Microsoft Entra Cloud Sync is specifically designed to synchronize on-premises AD users and groups into Entra ID using lightweight Microsoft agents.

Decision point

If the requirement is to keep AD objects synchronized and managed as Microsoft hybrid identities, Entra Connect or Cloud Sync remains the more natural Microsoft path.

Okta can overlap with Cloud Sync for some provisioning outcomes.
Okta does not fully replace Cloud Sync as Microsoft's AD-to-Entra synchronization engine.

11. Can Okta Fully Replace Intune?

Short answer: No. Okta is an identity platform; Intune is endpoint and application management.

Okta role Access decision support

Okta can help with authentication, MFA, SSO, sign-on policies, and in some designs device context through integrations with device management or security tools.

Intune role Device management

Intune provides MDM, MAM, enrollment, compliance policies, configuration profiles, app deployment, Windows Autopilot, update rings, BitLocker policy, security baselines, app protection policies, remote wipe, and cross-platform endpoint management.

Practical distinction

Okta can help decide whether a user should access an app. Intune manages whether the device and applications are configured, compliant, protected, and recoverable.

Okta can help decide access based on identity and device context.
Intune manages and secures devices and applications.
Okta does not fully replace Intune.

12. Can Okta Perform Writeback?

Short answer: Yes, in supported scenarios. The important question is writeback of what, from where, and to which target.

Writeback scenario Typical answer Notes
Okta to AD user updates Supported in some designs Okta can create or update users in AD when lifecycle management and AD integration are configured appropriately.
Okta to AD password reset or change Supported with AD integration Password reset or change can be written to AD depending on Okta AD Agent, delegated authentication, and password policy configuration.
Okta to SaaS applications Common Provisioning, profile updates, app assignment, and deactivation are commonly handled through SCIM or application APIs.
Okta to Entra ID / Microsoft 365 Depends Okta can provision some user and attribute flows, but this does not equal every Microsoft hybrid writeback or synchronization capability.
Entra ID to AD writeback Usually Microsoft sync This is normally handled by Entra Connect or Cloud Sync features, depending on the specific writeback requirement.
Discovery point: Never ask only whether Okta does writeback. Ask which object, which attribute, which direction, which source of authority, and which operational process owns the writeback.
Yes, Okta can perform writeback in supported integrations, especially to AD and SaaS apps.
No, Okta does not provide every Microsoft hybrid writeback capability that Entra Connect or Cloud Sync may provide.

13. Discovery Checklist

Use this checklist during client workshops or technical discovery sessions.

Identity source

  • What system creates the employee identity first?
  • Is AD, Okta, Entra ID, HR, or IGA the authoritative source?
  • Which attributes are mastered where?
  • Which system owns username, email address, manager, department, title, and employee status?

Directory integration

  • Is the Okta AD Agent installed?
  • How many AD domains and forests are connected?
  • Are users and groups imported from AD into Okta?
  • Are AD passwords delegated to Okta or synchronized through another method?

Entra ID integration

  • Is Entra Connect or Cloud Sync in use?
  • Do Entra ID users show as cloud-only or synchronized from on-premises?
  • Is Microsoft 365 authentication federated to Okta?
  • Are users cloud-only, synced, or mixed?
  • Are group-based licensing and Conditional Access policies active?

Authentication

  • Where does the user enter credentials?
  • Does Entra ID redirect login to Okta?
  • Are password hash sync, pass-through authentication, or federation configured?
  • How are non-browser and legacy authentication scenarios handled?

MFA and access policy

  • Is MFA enforced by Okta, Microsoft, or both?
  • Are Entra Conditional Access policies active?
  • Are device compliance, location, risk, or network rules used?
  • Are there different policies for admins, contractors, and frontline users?

Lifecycle

  • How is a new user created?
  • How are role changes and department changes handled?
  • How quickly is access removed when a user leaves?
  • Which system is responsible for disabling accounts and revoking sessions?

14. Risks and Watch Points

Risk What to check Client impact
Risk
Unclear ownership		 Document which platform owns each identity attribute, each lifecycle step, and each access decision. Conflicting updates, stale attributes, failed provisioning, and slow incident resolution.
Risk
Duplicated MFA		 Compare Okta sign-on policies with Entra Conditional Access, security defaults, per-user MFA, and authentication method policies. Users may receive multiple prompts or inconsistent access decisions.
Risk
Leaver gaps		 Trace the offboarding process from HR or AD through Okta, Entra ID, Microsoft 365, and business applications. Former users may retain access longer than expected, especially in applications not provisioned by Okta.
Risk
Group sprawl		 Identify whether access is controlled by AD groups, Okta groups, Entra groups, dynamic groups, or app-specific roles. Entitlements become hard to audit, automate, and explain to security stakeholders.
Risk
Federation dependency		 Confirm break-glass accounts, emergency access procedures, federation certificate expiry, Okta availability dependencies, and rollback options. Microsoft 365 access can be impacted if federation, certificates, network paths, or Okta services are unavailable.
Risk
Legacy authentication		 Check whether basic authentication, legacy protocols, service accounts, or older clients bypass expected MFA and Conditional Access controls. Security controls may be weaker than the apparent design suggests.
Risk
Premature Entra Connect removal		 Confirm user source, synchronized object status, Exchange Hybrid, device join, password writeback, group writeback, and provisioning replacement before changing sync. Removing sync without a replacement model can break user updates, licensing, Exchange attributes, offboarding, and Microsoft 365 access operations.
High priority: Always validate emergency access accounts in Entra ID. They should not depend on Okta federation, normal MFA flows, or a single administrative team.

15. Client Questions

These questions are designed to reveal the actual operating model instead of only the intended architecture.

Identity ownership

  1. What is the authoritative source for user identities?
  2. Which attributes are mastered in AD, Okta, Entra ID, HR, or another platform?
  3. Is Okta importing users from AD, provisioning users to AD, or both?
  4. Is Okta only used for SSO/MFA, or does it provision users into Entra ID?
  5. Are there cloud-only users in Entra ID or Okta-only users outside AD?
  6. If AD is retired, will Okta or HR become the authoritative source for employee identity?

Authentication and MFA

  1. Is Microsoft 365 authentication federated to Okta?
  2. Where is MFA enforced today: Okta, Entra ID, or both?
  3. Are Entra Conditional Access policies active?
  4. How are administrators, service accounts, and emergency accounts handled?

Synchronization and provisioning

  1. Do you use Entra Connect, Cloud Sync, or another synchronization tool?
  2. Do Entra ID users show as synchronized from on-premises?
  3. Which applications are provisioned by Okta?
  4. If Entra Connect is removed, what system will create, update, disable, and license users in Entra ID?
  5. Which applications still require manual access management?
  6. How are licenses assigned in Microsoft 365?

Operations and governance

  1. Which groups control application access?
  2. What happens when a user changes department or role?
  3. What happens when a user leaves the company?
  4. Which downstream systems must receive deactivation events from Okta?
  5. Who owns identity operations: IAM, AD team, Microsoft 365 team, Security, or HR IT?
